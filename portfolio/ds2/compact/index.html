II. NARRATIVE

What we accomplished, what I contributed

During this group project, we designed a system to transfer luggage from a platform to another through Autodesk Inventor, while taking in software input on where to place the luggage with Python. We then built and tested a prototype using 3D printing and mechanical actuators.

Project

Problem:
Oct. 3, 2024
  • Software
    • We were tasked with creating modular programs using Python to perform data validation and analysis.
  • Hardware
    • We were tasked with designing a mechanism to transport luggage from Point A to Point B (Fig. 2).

Fig. 2: Diagram of the challenge. Note restriction line.

Problem Analysis:
Oct. 3, 2024
  • Objectives
    • Effective at delivering luggage
    • Consistent under repeated use
    • Deliver luggage safely
  • Functions
    • Scan barcode to determine delivery location
    • Move luggage to right location
    • Process fleet and passenger data
  • Constraints
    • Material effective
    • Timely delivery rate
    • No part may pass the restriction line (Fig. 2)
    • Only moving parts are the two actuators provided: rotary and linear

  • Final Problem Statement
    • The problem is that a meaningful proportion of luggage at airports is lost in transit. We aim to create a system that delivers bags to their correct locations based on airport regulations. This solution aims to reduce the amount of bags that are lost at airports by using automation.

Fig. 3: My 'bucket' design. Created through three days of iterative development. The 'bucket', attached to linear actuator, rotates using the rotary actuator.

Brainstorming:
Oct. 4, 2024 - Oct. 10, 2024

Two initial solutions:

  • The 'elbow'
  • The 'bucket'

The 'bucket' was my idea (Fig. 3, click to switch images); functional design features include:

  • A smooth ramped bucket for easy delivery
  • Weighted base to lower stress on screws
  • Hollowed components to lower rotary torque and increase control
  • Extended guards for smoother angular movement
The Solution:
Oct. 11, 2024 - Nov. 6, 2024

Ended up with 'elbow' instead, as my design was too complicated to 3D print. More refinements were made, and each member modeled a part. I modeled a support for the rotary actuator. During this time, 3D printing opportunities began: I attended most of these as representatives for our group, printing parts.

Code:
Oct. 31, 2024 - Nov. 25, 2024

I was assigned the role of Software Lead. One function was to be completed by each member; three were to be completed by the team. I completed two of the three team functions, and finished my function time_delay.py. I took iterative feedback from the team in this process.

Fig. 4: The moment I notified my team that I got the graphical_teamID.py module running.

You can find the full attributed code on GitHub here. After the code was finished, I took all modules and executed these steps:

  • Collected all Python modules and performed unit tests.
  • Standardized all I/O methods.
  • Made code readable + added comments.
  • Linked modules through an external _config.py file.
  • Retrieved and completed the final team function, graphical_teamID.py.
  • Linked all modules to one another as inputs.
  • Tested and optimized graphical_teamID.py.
  • Performed final integration tests for commiting to Github.

I might even say that I practically rewrote a significant amount of code to fix bugs, solve edge cases and inefficient chunks, and refactored the whole existing base of graphical_teamID.py, since it was outdated.

Evaluation:
Dec. 2, 2024
Total span: Oct. 3, 2024 - Dec. 2, 2024 (60 days)

We finished up our printing and I briefed the team on my changes to their code and final questions on how the code worked. I also assisted on small physical testing of the mechanism. We unfortunately suffered 3D printing issues because of slicing errors and improper supports, meaning we had no testing time for the full mechanism.

Luckily, on the day of the evaluation, when we arrived half an hour earlier for our allotted setup time, the solution worked wonderfully. See Fig. 5 for an image of the full assembled solution on the deployment pad. Tap on the image to toggle extended and retracted forms.

Reflection

ON CAD

I feel like I could have done some things better. I think that I was not proactive enough during the CAD section of the project. Another group member took it upon themselves to link together all of the individual components into an assembly and fix the conflicts and errors that arose, like I did with the code. The group member did not ask the rest of the team for help, but I feel like if I had made my willingness and ability to work with CAD clearer to the group, the member would have let me help them with the assembling. I potentially could have found and fixed some of the issues that plagued our 3D printing process. That would have led to us having the testing time that we missed, which could have given us the ability to do better.

However, I feel like I learned a lot about the entire pipeline of 3D printing. I familiarized myself with PrusaSlicer, the integration of assembly parts in AutoDesk Inventor, how to use motion constraints and animate parts, and the creation of engineering drawings and exploded views.

ON PYTHON

Additionally, I feel as if I did not sufficiently ask my teammates for feedback. While I posted occasional progress checks and git commited my code regularly, I think that I should have been in active communication with my team members while doing the major refactoring and linking. I wonder if the code could have been even more compact, even more efficient, even more readable had I been coding live with my team?

All of my hobby projects have some form of rigor or linkage between files, but not to this level. I learned some new things about Python: while I had casual familiarity with using Python files as modules, reading settings and configuration from a .py file instead of a .yaml or .json file, and working on code with others through git, this project was my first time putting these principles into action on a real-world codebase. I feel like I gained a greater appreciation for source control, unit tests, and integration tests. This appreciation was especially evident since I was running and working with code written by others, without guarantees of standardization or proper functionality.
